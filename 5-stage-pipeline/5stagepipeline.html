Pipeline Demonstration

There are many ways to 'pipeline' a unit. A 'classic' example is a processor divided into 5 stages (as discussed in the intro videos). This is a realistic division which is also useful to illustrate some problems with pipelines and ways to alleviate these problems.

Fetch
Read an instruction as indicated by PC, including instruction memory access; increment PC.
Decode
Work out what the instruction is to do; read any register operands.
Execute
Perform any calculation, including any address derivation.
This stage also determines the branch destination.
Memory
Perform any data transfer for loads and stores; just a placeholder for purely internal operations.
Writeback
Retire results to the register file for reuse.

Every instruction can be executed by progressing along this pipeline although some circumstances ("dependencies") may introduce delays to a perfectly smooth flow.

Experiments

  1. Send non-dependent instructions
  2. Add a jump: observe pipeline flush
  3. Add some (close) dependencies: observe forwarding
  4. Add a dependency on an immediately preceding load: observe stall
  5. Add a dependency on a store: no stall needed

Note that this pipeline assumes assumes that instruction fetches and data transfers can take place simultaneously: i.e. a Harvard architecture.

The potentially dependent 'slot' following a load is called a delay slot. Deeper pipelines may have several consecutive delay slots.

Note that the load latency penalty can be avoided if a useful, non-dependent instruction follows a load. Most compilers will spot dependencies and (if possible) reorder the code to separate dependent instructions.
High-performance processors will detect dependencies at instruction-decode time and reorder the instruction stream to fill a delay slot if they can.
