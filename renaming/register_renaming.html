Register renaming

Register renaming is a more flexible mechanism for reordering (as in sorting back into order those which have got out of order) instructions and enabling clean handling of hazards.

Renaming separates the name (address) of the physical register from the name the programmer is using for it. This is a dynamic process. Each time the code wants to write to a register a fresh physical register is assigned. This means that more registers are implemented than the programmer/compiler sees. (The demonstration below has far fewer registers than a RISC processor.)

Note: for legibility and comparison purposes the users' register names have been left in the processing stages (this information isn't needed by the machine.

The register mappings – as seen at the issue stage – can be recorded each time there is the possibility of an exception (such as when loads or stores are issued); if an exception occurs this can then be restored. Physical registers are recycled when their old values are no longer of any use, which is when all preceding operations have completed or are guaranteed to complete correctly.

The possibility of superscalar issue should be fairly clear, here. A load and a move can be issued simultaneously when there are no dependencies = which there aren't between the pairs of operations as written.

There will be more physical registers than register names in the ISA (Instruction Set Architecture). Given sufficient physical registers it is possible to support multithreading (discussed in Week 9); each thread has an exclusive dynamic selection of the available physical registers which allows the instruction issuer to choose instructions from a mixture of threads on a cycle-by-cycle basis. Each thread has a separate mapping for its registers so all the R0s (etc.) map to unique physical registers.
