WAW Hazard & Reorder Buffer

The ‘classic’ five stage pipeline model includes a ‘data memory’ stage: internal operations (the majority) traverse this without additional processing. A more sophisticated structure can introduce multiple, parallel processing pipelines. These may not be the same length.

Here is illustrated a WAW hazard where execution of one operation could overtake another and the register modifications could happen in a different order from the intended one. This would be a Bad Thing.

There are various ways to overcome this problem. The one illustrated is a reorder buffer. When issued the operations are tagged with a sequence number; this acts as a position in the reorder buffer where the result is placed. The issue stage notes the destination so it can use this as a source for recent results Once the result is in the reorder buffer it validates the reorder buffer slot and can be forwarded to subsequent operations as necessary, although only to operations later in the object code.

To avoid running out of reorder buffer space the results are retired to the register file. This is done in order, oldest first. Only validated results from otherwise completed operations can be retired. When a result it retired the slot becomes available for reuse which is done cyclically.
Not all instructions produce register results (branches for example) but if there are concerns about retiring too slowly - for example to catch up after having had to wait - more than one result could be written in each cycle.

If parallel processing pipelines exist it may be possible to issue more than one instruction in a single cycle. Instructions can even be issued out of order. Provided that their reorder buffer sequence reflects the written order of the operations this is not going to affect the programmers' view of execution - except it might run a bit faster.

Results in a reorder buffer may be reused for later operations but they are still temporary and speculative. In nearly every case they will be correct. Occasionally, though, something will go wrong. The most common such exception will be a memory fault which triggers paging.

This is a hard problem, since what is wanted is the memory to work and the application to continue but, to reach that point the processor needs to service the exception, organise the paging (via the O.S.) and, in all likelihood, goes away to run other applications whilst waiting for the disk. When it returns it must be able to resume as if nothing had happened. For example, if a register value had been incremented after the load it must not be incremented again!

There are two, basic approaches:

  • Record everything which has already been done and don't do these (again) when returning.
  • Discard any changes which should have followed the exception point and restart from the load (or store) which caused the exception.

In principle the first option saves a little (repeat) processing but it is much more complex to implement than the second. Anyway, such exceptions are rare and already expensive when they happen so a little extra processing cost is not significant.

A reorder buffer makes the second option easy. The failed load (or store) is noted in its appropriate reorder buffer slot and all the (speculative) results behind that are discarded. When all results up to that point have been retired the processor is in a ‘clean’ state with regard to the instructions written by the programmer/compiler. An exception return to rerun the failed load (or store) will exactly reflect what was intended.

In the interactive example, if the first load aborted it would be signalled at cycle 4 when S0 is filled. Notice that R1 has already been changed and used but no permanent changes have reached the register file; they can all be safely discarded.
